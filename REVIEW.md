## 一、项目概览

DeepHarness Desktop 是一个 AI 编码助手桌面应用，采用 **React SPA + Tauri v2** 架构，支持用户选择多种 AI 智能体（OpenCode、Claude Code、Cursor Agent、Codex、自定义）进行多轮编码对话，并提供会话管理、任务管理、文件变更追踪、智能体配置等完整功能。

### 核心指标

| 维度 | 数据 |
|------|------|
| TypeScript 前端文件 | ~60 个源文件（含 50+ shadcn/ui 组件） |
| Rust 后端文件 | ~50 个源文件（含 3 个 workspace crate） |
| 数据库表 | 10 张 SQLite 表 |
| 页面路由 | 4 个（登录、选智能体、工作区、404） |
| Zustand Store | 5 个 |
| 测试文件 | 1 个（5 个测试用例） |
| 已记录缺陷 | 8 个 |
| 设计文档 | DESIGN.md + 14 份超能力设计文档 |

## 二、架构审查

### 2.1 总体架构

```
┌──────────────────────────────────────────────────┐
│                  Web 前端 (React SPA)              │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ Pages    │ │ Components│ │ Zustand Stores   │  │
│  │ (4个)    │ │ (60+)    │ │ (5个)            │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
│         │            │                │           │
│         └────────────┼────────────────┘           │
│                      │ WebSocket (JSON-RPC 2.0)   │
├──────────────────────┼────────────────────────────┤
│               Tauri Bridge (IPC)                  │
│                      │                            │
├──────────────────────┼────────────────────────────┤
│              Rust 后端 (Tauri)                     │
│  ┌──────────┐ ┌─────┴─────┐ ┌──────────────────┐ │
│  │ Gateway  │ │  Services │ │ Agent System     │ │
│  │ (WS+JSON)│ │ (DB/Agent)│ │ (3 crates)       │ │
│  └──────────┘ └───────────┘ └──────────────────┘ │
│         │            │                │           │
│         └────────────┼────────────────┘           │
│                      ▼                            │
│               SQLite + 文件系统                    │
└──────────────────────────────────────────────────┘
```

### 2.2 架构评估

**优势：**

1. **前后端通过 WebSocket JSON-RPC 通信** — 相比 Tauri 原生 invoke，WebSocket 网关提供了统一的异步消息通道，支持服务端推送和流式响应
2. **Agent 插件系统设计良好** — 3 层 crate 抽象（core → runtime → plugin）实现了清晰的关注点分离，便于扩展新智能体
3. **数据库适配层（双模式）** — Mock/Tauri 自动切换，支持浏览器开发环境独立运行
4. **流式处理架构** — SSE 优先 + HTTP 兜底 + 模拟流式三种策略，容错设计周全

**不足：**

1. **`main.rs` 达 947 行** — 承担了 DB 初始化、WebSocket 服务器、Agent 服务、插件注册、命令注册、窗口配置等多重职责，严重违反单一职责原则
2. **流式处理回退层（模拟流式）** 将文本切分为 1-3 字符片段、15-40ms 延迟发送，这一策略在逻辑上可行但存在用户体验风险（延迟不自然、可能产生闪烁）
3. **前端大型组件未拆分** — ChatPanel (1046 行)、WorkspacePage (960 行)、SettingsDialog (956 行)、LeftPanel (897 行)，均超出合理单文件规模

### 2.3 数据流分析

**消息发送主路径：**
```
用户输入 → chatStore.sendMessage() → websocketStore.send()
→ Rust Gateway (JSON-RPC) → streaming.rs → opencode_service
→ HTTP POST opencode serve → SSE events → Gateway → 前端
```

**关键风险点**：
- 消息链路过长（7 跳），任一环节故障都可能导致消息丢失
- SSE 断连时依赖 HTTP 轮询和模拟流式，存在时序竞态条件

## 三、代码质量审查

### 3.1 TypeScript 前端

| 文件 | 行数 | 问题等级 |
|------|------|----------|
| ChatPanel.tsx | 1046 | ⚠️ 严重过大 |
| WorkspacePage.tsx | 960 | ⚠️ 严重过大 |
| SettingsDialog.tsx | 956 | ⚠️ 严重过大 |
| LeftPanel.tsx | 897 | ⚠️ 严重过大 |
| RightPanel.tsx | 369 | ⚠️ 偏大 |
| sidebar.tsx | 771 | ⚠️ 偏大（shadcn 组件可接受） |

**具体问题：**

1. **ChatPanel.tsx 1046 行** — 融合了消息列表渲染、输入框、流式文本处理、交互响应等多个职责，建议拆分为 MessageList、MessageInput、StreamingText 等子组件
2. **WorkspacePage.tsx 960 行** — 承担了三栏布局编排、Agent 实例管理、会话切换、WebSocket 连接等多重职责，建议将布局逻辑与业务逻辑分离
3. **SettingsDialog.tsx 956 行** — 4 个 Tab（通用/技能/提示词/工程规范）挤在一个文件中，建议每个 Tab 独立为组件
4. **LeftPanel.tsx 897 行** — 文件树 + 会话历史两个核心功能耦合在同一个文件中

### 3.2 Rust 后端

| 文件 | 行数 | 评价 |
|------|------|------|
| main.rs | 947 | 承担过多职责 |
| streaming.rs | 429 | 流式处理逻辑集中 |
| opencode_service.rs | 351 | 合理 |
| db_service.rs | 295 | 合理 |
| agent_service.rs | ~100 | 合理 |

**具体问题：**

1. **`main.rs` 应为引导文件**，不应承担模块级逻辑。建议拆分为 `setup_db.rs`、`setup_websocket.rs`、`setup_plugins.rs`、`setup_commands.rs`
2. **Rust 工作区缺乏统一的 lint 配置** — 无 `clippy.toml`、`rustfmt.toml`、`[lints]` 段
3. **无 `#[deny(warnings)]`**，编译警告不会被阻断
